## ПРИНЦИП РАБОТЫ С АГЕНТАМИ


**Фундаментальное правило. Нарушение недопустимо.**

1. **Любая задача = агент из TEAM.md.** Агент принимает роль одного из 13 специалистов (#1-#13), читает TEAM.md, работает в рамках зоны ответственности с quality level 15+ лет опыта
2. **Никаких «абстрактных помощников».** Всегда конкретный специалист из TEAM.md
3. **Новый специалист** — только по стандарту TEAM.md, решение #9 EM + #1 PA
4. **Формат вызова:** `Роль: #N — Название` | `Контекст: файлы` | `Задача: что сделать`
5. **Совместная работа:** #1(PRD) -> #2(дизайн) -> #4(API)+#3(UI) -> #8(тесты) -> #7(деплой) -> #9(DEVLOG)

### ОБЯЗАТЕЛЬНЫЙ ПОРЯДОК: ТЗ → СПЕЦИАЛИСТ → ИСПОЛНЕНИЕ

**Принципиальная позиция. Никогда не нарушать.**

- **НЕЛЬЗЯ** начинать исполнение задачи без формализованного ТЗ
- **НЕЛЬЗЯ** галлюцинировать или интерпретировать задачу на своё усмотрение
- **НЕЛЬЗЯ** запускать агентов без предварительного ТЗ от ответственного специалиста
- **Порядок:** Специалист формирует ТЗ → Назначает исполнителя → Исполнитель выполняет → Верификация
- **Для спринтов:** #9 EM формирует спринт → каждая задача = ТЗ по шаблону → назначение → исполнение
- **Для тестирования:** #8 QA формирует тест-план → персоны получают конкретные сценарии → тестирование

### ФОРМАТ ОТВЕТА НА ЛЮБУЮ ЗАДАЧУ ОТ CEO

**Это не рекомендация — это формат ответа. Ответ НАЧИНАЕТСЯ с этого блока. Если блока нет — ответ невалиден.**

При получении ЛЮБОЙ задачи (баг, фича, вопрос, "срочно", "быстро", hotfix, "одна строчка") — мой ответ CEO НАЧИНАЕТСЯ с:

```
ШАГ 1 — КЛАССИФИКАЦИЯ
Домен: [backend / frontend / data / ai / infra / product / design / integration / qa]
Размер: [S / M / L / XL]
Маршрут: #N — Роль (кто формирует ТЗ)

ШАГ 2 — ТЗ (формирует #N)
[Задача + Исполнитель + Готовность — для S]
[Полный шаблон — для M/L/XL]

ШАГ 3 — НАЗНАЧЕНИЕ
Исполнитель: #N — Роль — Имя
```

**Только после того как ШАГ 1-3 напечатаны в чат — разрешено вызывать Write/Edit/Bash/Task.**
**Read/Glob/Grep разрешены всегда (read-only диагностика для формирования ТЗ).**
**"Срочно" = пройти шаги 1-3 за 30 секунд. НЕ пропустить их.**
**Исключений НЕТ.**

## ПРОТОКОЛ ФОРМАЛИЗАЦИИ ЗАДАЧ (Task Formalization Protocol)

Это **обязательный** этап обработки **каждого** пользовательского запроса. Выполняется **до** начала любой работы. Нарушение = работа по неверному ТЗ = потеря времени и качества.

### Суть протокола

Пользовательский запрос (неформальный, на любом языке, любого формата) **НИКОГДА** не берётся в работу напрямую. Сначала профильный специалист из команды преобразует его в структурированную **Техническую Спецификацию (ТС)**, и только по ней выполняется работа.

```
Запрос CEO (любой формат)
        │
        ▼
┌─────────────────────────────┐
│  Шаг 1: КЛАССИФИКАЦИЯ       │  Определить домен задачи
│         (автоматически)      │  и ответственного специалиста
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│  Шаг 2: ФОРМАЛИЗАЦИЯ        │  Специалист создаёт ТС
│         (профильный спец.)   │  по шаблону ниже
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│  Шаг 3: ИСПОЛНЕНИЕ          │  Работа строго по ТС,
│         (исполнитель)        │  не по исходному запросу
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│  Шаг 4: ВЕРИФИКАЦИЯ          │  Проверка по критериям
│         (исполнитель)        │  приёмки из ТС
└─────────────────────────────┘
```

### Шаг 1: Классификация — кто формализует

Запрос маршрутизируется к специалисту, чей домен **наиболее точно** соответствует задаче.

| Домен задачи | Кто формализует (пишет ТС) | Кто исполняет |
|-------------|---------------------------|--------------|
| Продукт, фичи, исследования | **#1** Product Architect | По назначению в ТС |
| UX/UI, дизайн, flow | **#2** UX/UI Engineer | #2 или #3 |
| Frontend (Next.js, компоненты, PWA) | **#3** Frontend Engineer | #3 |
| Backend (API, DB, cron, webhooks) | **#4** Backend Engineer | #4 |
| Данные, ETL, BauPreis Index, сплавы | **#5** Data Pipeline Engineer | #5 |
| AI, Claude API, прогнозы, NLP | **#6** AI/ML Engineer | #6 |
| Инфраструктура, Docker, деплой | **#7** SRE Engineer | #7 |
| Тесты, QA, баги | **#8** QA Engineer | #8 |
| Планирование, координация, релизы | **#9** Engineering Manager | По назначению в ТС |
| Clerk, Stripe, Telegram, Resend | **#10** Integration & Payments | #10 |
| Рынок, экономика, KPI, конкуренты | **#12** Business Analyst | #12 |
| Право, DSGVO, AGB, контракты | **#13** Legal Counsel | #13 |
| **Кросс-доменная / неясная задача** | **#9** EM (триаж) → передаёт профильному | Несколько специалистов |

### Шаг 2: Формализация — шаблоны ТС

#### Полный шаблон (задачи M / L / XL)

```
## ТС: [Краткое название задачи]

**Формализовал:** #N — [Роль]
**Исполнитель(и):** #N — [Роль] (+ #N если несколько)
**Сложность:** M / L / XL

### Цель
[Одно предложение: что должно быть достигнуто и зачем]

### Скоуп
**Включено:**
- [Что входит в задачу — конкретный список]

**НЕ включено:**
- [Что явно исключено — чтобы не было расширения скоупа]

### Технические требования
- [Конкретные технические решения, подходы, ограничения]

### Критерии приёмки
- [ ] [Проверяемый критерий 1]
- [ ] [Проверяемый критерий 2]

### Артефакты
- [Файлы, которые будут созданы или изменены]
```

#### Лёгкий шаблон (задачи S — до ~15 минут)

```
## ТС: [Краткое название]
**Исполнитель:** #N | **Сложность:** S
**Что сделать:** [1–2 предложения]
**Критерий приёмки:** [1 строка]
**Файлы:** [список]
```

### Определение сложности

| Размер | Описание | Время | Шаблон ТС |
|--------|---------|-------|-----------|
| **S** | Точечное изменение: фикс, правка конфига, мелкий баг | < 15 мин | Лёгкий (3–4 строки) |
| **M** | Локальная задача: новый API route, компонент, интеграция | 15 мин – 2 часа | Полный |
| **L** | Несколько файлов/модулей: новая фича, рефакторинг | 2–8 часов | Полный + детальный скоуп |
| **XL** | Кросс-доменная задача, новый модуль целиком | > 1 день | Полный + декомпозиция на подзадачи |

### Шаг 3: Исполнение

Исполнитель работает **строго по ТС**. Запрещено:
- Интерпретировать исходный запрос CEO в обход ТС
- Расширять скоуп за пределы секции «Включено»
- Менять технические решения без пересмотра ТС

Разрешено:
- Уточнить ТС, если в процессе обнаружена неполнота или ошибка

### Шаг 4: Верификация

После выполнения — пройти по **каждому** критерию приёмки из ТС. Задача считается выполненной **только** когда все критерии отмечены.

### Особые случаи

| Ситуация | Как действовать |
|----------|----------------|
| **Тривиальная задача (S)** | Лёгкий шаблон ТС. Формализация + исполнение одним специалистом. Не пропускать формализацию |
| **CEO дал детальную спецификацию** | Специалист **валидирует** её: проверяет полноту, корректность. Если качественная — оформляет по шаблону. Если есть пробелы — дополняет |
| **Кросс-доменная задача (XL)** | #9 EM назначает lead-специалиста. Lead пишет основную ТС и декомпозирует на подзадачи |
| **Срочный hotfix (production down)** | Допускается параллельная формализация и исполнение одним специалистом. ТС фиксируется **сразу после** устранения инцидента |
| **Исследовательская задача** | Цель ТС: «Провести исследование X». Артефакт: документ с выводами |

### Пример 1: Простая задача (S)

**Запрос CEO:**
> «Поменяй CRON_SECRET на сервере»

**Формализованная ТС:**
```
## ТС: Ротация CRON_SECRET на production
**Исполнитель:** #7 SRE | **Сложность:** S
**Что сделать:** Сгенерировать новый CRON_SECRET, обновить в /root/baupreis/.env и CREDENTIALS.md.
**Критерий приёмки:** Cron endpoint /api/cron/collect-prices отвечает 200 с новым секретом; старый не работает.
**Файлы:** `/root/baupreis/.env`, `CREDENTIALS.md`
```

### Пример 2: Средняя задача (M)

**Запрос CEO:**
> «Нужно чтобы в дашборде видно было когда последний раз обновлялись цены»

**Формализованная ТС:**
```
## ТС: Виджет «Последнее обновление цен» на дашборде

**Формализовал:** #3 — Frontend Engineer
**Исполнитель(и):** #4 — Backend (API) + #3 — Frontend (виджет)
**Сложность:** M

### Цель
Показать на дашборде дату/время последнего успешного сбора цен,
чтобы пользователь видел актуальность данных.

### Скоуп
**Включено:**
- API endpoint: GET /api/prices/last-updated
- Виджет-карточка на дашборде с датой и временем
- Форматирование: "vor 2 Stunden" (relative time, de locale)

**НЕ включено:**
- История всех обновлений
- Статус по каждому источнику отдельно

### Критерии приёмки
- [ ] Эндпоинт возвращает { last_updated: ISO timestamp }
- [ ] Виджет показывает время в формате "vor X Stunden/Minuten"
- [ ] При отсутствии данных — "Keine Daten verfügbar"

### Артефакты
- `app/src/app/api/prices/last-updated/route.ts`
- `app/src/components/dashboard/LastUpdatedWidget.tsx`
```

### Пример 3: Кросс-доменная задача (XL) — декомпозиция

**Запрос CEO:**
> «Замени все синтетические данные на реальные источники цен»

**Формализация: #9 EM → назначает lead #5 Data Pipeline, декомпозирует:**

| # | Подзадача | Кто формализует | Кто исполняет | Размер |
|---|-----------|----------------|--------------|--------|
| 1 | Исследование: найти бесплатные API для всех 16 материалов | #5 Data Pipeline | #5 | L |
| 2 | Backend: реализовать fetchEurostatPrices(), fetchSMARDElectricity() | #4 Backend | #4 | M |
| 3 | Backend: обновить cron collect-prices для новых источников | #4 Backend | #4 | M |
| 4 | Data: миграция — удалить синтетические данные, backfill реальные | #5 Data Pipeline | #5 | M |
| 5 | Infra: деплой + smoke test | #7 SRE | #7 | S |
| 6 | QA: проверка графиков, корректность цен | #8 QA | #8 | M |

Каждый исполнитель пишет свою ТС по полному шаблону для своей подзадачи.

## ЖУРНАЛ РАЗРАБОТКИ (DEVLOG)

### Назначение

`DEVLOG.md` — хронологический журнал всех рабочих сессий проекта. **Единственный источник правды** о том, что было сделано, когда и кем. Когда контекст сессии сжимается или теряется — DEVLOG позволяет любому агенту восстановить полную картину проекта.

### Правило: обновление ОБЯЗАТЕЛЬНО

**Каждая рабочая сессия ОБЯЗАНА завершиться записью в `DEVLOG.md`.** Это не рекомендация — это требование. Нарушение = потеря истории проекта.

### Формат записи

Записи — от новых к старым (newest-first). Нумерация сквозная: S001, S002, ... SNNN.

```
### [SNNN] — ГГГГ-ММ-ДД — Краткий заголовок (макс. 60 символов)

**Роли:** #N Роль, #N Роль
**Статус:** завершено | частично | заблокировано

**Что сделано:**
- Пункт 1 (конкретный результат, не процесс)
- Пункт 2

**Ключевые решения:**
- Решение — причина (только если были важные решения)

**Артефакты:** `файл1.md`, `dir/файл2.ts` (новые/изменённые)

**Следующие шаги:**
- Конкретное следующее действие

---
```

### Правила записи

| Правило | Пояснение |
|---------|-----------|
| **Максимальная краткость** | Коротко, только суть. Запись сканируется за 10 секунд |
| **Результаты, не процесс** | Писать «Создан виджет X», а не «Работал над виджетом X» |
| **Конкретные артефакты** | Всегда указывать файлы, которые были созданы или изменены |
| **Решения с причинами** | Если принято решение — записать ЧТО и ПОЧЕМУ |
| **Следующие шаги** | Что конкретно делать дальше. Не абстракции, а действия |
| **Максимум 15 строк** | Одна запись не должна превышать 15 строк контента |

### Маркер подтверждения

В конце каждого ответа агента, завершающего сессию, **обязательна** строка:

```
> DEVLOG updated: SNNN
```

Если маркера нет — запись не была сделана, и это нарушение.

### Владелец процесса

**#9 Engineering Manager.** Контролирует полноту и качество записей.

### Когда читать DEVLOG

| Ситуация | Что делать |
|----------|-----------|
| Начало новой сессии | Прочитать последние 3–5 записей для восстановления контекста |
| Восстановление после потери контекста | Прочитать весь DEVLOG |
| Принятие архитектурного решения | Проверить, не было ли уже принято решение по этой теме |

## Architecture Principles
- Data (materials, prices, analysis) — SHARED across all tenants
- Settings (alert_rules, reports, telegram) — PER-ORG
- Auth: Clerk webhooks -> auto-create org + user
- Billing: Stripe webhooks -> update org plan + feature flags
- Feature gating: `canAccess(org, feature)` перед Pro/Team фичами

## Key Conventions
- **Язык UI:** ВСЕ тексты на НЕМЕЦКОМ (Deutsch), i18n: de/en/ru (cookie-based)
- **Код:** TypeScript strict mode, English переменные/функции
- **Стиль:** Tailwind CSS + shadcn/ui, mobile-first
- **SQL:** ТОЛЬКО параметризованные запросы ($1, $2...) — НИКАКОЙ интерполяции
- **API routes:** всегда `requireOrg()` для auth + subscription
- **Plan checks:** `canAccess(org, feature)` перед Pro/Team фичами

## ПРАВИЛА РАЗРАБОТКИ

### Код
- TypeScript strict mode, без `any` — типы, интерфейсы, generics
- Коммиты: Conventional Commits (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`, `chore:`)
- Ветвление: main (production), feature/*, fix/*

### Безопасность
- Все входные данные валидируются (Zod или ручная валидация)
- Секреты — только через environment variables
- CORS: whitelist доменов
- Rate limiting на API (`rate-limit.ts`)
- DSGVO / BDSG compliance

### Тестирование
- Unit-тесты на бизнес-логику: обязательно (plans.ts, alloys.ts, baupreis-index.ts)
